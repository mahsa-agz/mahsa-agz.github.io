## The mental model: three trees

Almost everything in Git makes sense once you see it as three places your files can live. The working directory is what you edit. The staging area (also called the *index*) is the draft of your next snapshot ,  files you've marked as "include this." The repository is the permanent history of snapshots (`commits`), stored inside the `.git` folder.

## The everyday loop

Ninety percent of Git use is the same three-step cycle: see what's changed, stage what you want to keep, commit it with a message that explains *why*.

```bash
# 1. What changed?
git status

# 2. Stage the parts you want in the next snapshot
git add README.md src/parser.py

# 3. Save the snapshot with a message
git commit -m "Parser: handle empty input rows"
```

`git status` is the most under-used safety net in Git. Read it before every `add` and every `commit` ,  it tells you exactly what's about to move between trees.

## Commits are snapshots in a chain

A commit isn't a diff ,  it's a full snapshot of your project with a pointer to its parent. The history is just a chain of those snapshots. A **branch** is a movable label pointing at one of them. `HEAD` is the label that says "you are here."

## Branching off, merging back

Branches let you work without disturbing `main`. Create one, commit on it, then bring it back. Most teams call the long-lived line `main` and the short-lived ones *feature branches*.

```bash
git switch -c feature/login    # create + switch
# ...edit, add, commit...
git switch main
git merge feature/login        # bring it back
git branch -d feature/login    # delete the label
```

## When to make a branch ,  and what to call it

A new branch costs nothing. Make one any time the work is more than a typo. The branch name's job is to tell a teammate (or future-you) what's on it without having to look. Most teams use a small set of prefixes:

| Prefix | For | Example |
|---|---|---|
| `feature/` | new functionality | `feature/csv-export` |
| `fix/` | a bug fix | `fix/parser-empty-rows` |
| `hotfix/` | urgent production fix | `hotfix/login-500` |
| `refactor/` | restructure without changing behaviour | `refactor/extract-auth` |
| `chore/` | dependency bumps, build config, non-functional work | `chore/upgrade-react` |
| `docs/` | docs only | `docs/api-readme` |

Two rules that prevent 90% of branch pain:

1. **One branch, one purpose.** If you find yourself making unrelated commits, branch off again.
2. **Keep branches short-lived.** A week is normal; a month is risky; six months means the merge is going to hurt.

## Branching strategies, briefly

How a team uses branches is a *strategy*, and a few are standard:

| Strategy | How it works | Best for |
|---|---|---|
| **GitHub Flow** | `main` is always shippable. Every change is a short-lived branch off `main`, merged back via PR. | Most teams. Simple, low-ceremony. |
| **Git Flow** | `main` + `develop` + `feature/*` + `release/*` + `hotfix/*` branches. Releases are batched. | Versioned releases (mobile apps, libraries). |
| **Trunk-based** | Everyone commits to `main` directly or via very short-lived (hours) branches. Heavy use of feature flags. | High-frequency deploys, strong CI, experienced teams. |

Most teams use a flavour of GitHub Flow without ever giving it a name. Don't overthink it ,  pick the simplest one that fits how often you deploy.

## Merge vs. rebase

Both bring branches together. **Merge** preserves history exactly as it happened, with a merge commit. **Rebase** rewrites your branch so its commits sit on top of the new base ,  a straight line, no merge bubble.

## Conflicts: when they happen, how to fix them

A **merge conflict** happens when Git can't figure out how to combine two changes automatically. Two situations cause almost all of them:

1. **Same lines, different changes** ,  you and a teammate edited the same lines of the same file.
2. **Edit vs. delete** ,  you renamed or deleted a file a teammate also changed.

Git stops mid-operation and tells you which files are conflicted:

```
Auto-merging src/parser.py
CONFLICT (content): Merge conflict in src/parser.py
Automatic merge failed; fix conflicts and then commit the result.
```

Open the file. Git has left **conflict markers** in it:

```
<<<<<<< HEAD
def parse(line):
    return line.strip().split(",")
=======
def parse(line):
    return [x.strip() for x in line.split(",")]
>>>>>>> origin/main
```

Read it like this:

- Between `<<<<<<<` and `=======` is **your** version (the one on your branch, labelled `HEAD`).
- Between `=======` and `>>>>>>>` is **their** version (the branch you're trying to merge in, named after the `>>>>>>>`).

To resolve:

1. **Decide what the correct version is** ,  sometimes yours, sometimes theirs, often a hand-blended mix.
2. **Delete the `<<<<<<<`, `=======`, `>>>>>>>` markers** entirely. Don't leave any.
3. **Save the file.**
4. `git add <file>` ,  marks it as resolved.
5. `git commit` ,  finishes the merge. Git pre-fills a sensible message.

Most editors (VS Code, JetBrains, Vim with plugins) show a built-in conflict UI with "Accept current / Accept incoming / Accept both" buttons ,  much friendlier than editing the markers by hand.

If the merge is more trouble than it's worth and you want to back out:

```bash
git merge --abort      # back to the state before merging
git rebase --abort     # back to the state before rebasing
```

### How to avoid conflicts in the first place

- **Pull often.** Frequent small integrations beat one giant painful one.
- **Keep branches short-lived.** The longer your branch lives, the more `main` has drifted from under it.
- **Talk when you're in the same area.** A 30-second message in chat saves an hour of merge work.
- **Watch hotspots.** If every PR touches the same file, that file is too big ,  not a Git problem.

## Working with remotes

A *remote* is just another copy of the repo, usually on GitHub. The default name is `origin`. Three commands move work between you and it:

- `git fetch` ,  download new commits, don't touch your branch.
- `git pull` ,  fetch + merge into your current branch.
- `git push` ,  send your commits up.

Step by step:

1. **Update your local `main`.** `git switch main && git pull` so you start from the latest.
2. **Create a feature branch.** `git switch -c feature/your-thing`.
3. **Work in small commits.** Each commit does one thing and has a clear message ,  write *why*, not *what*.
4. **Push the branch.** `git push -u origin feature/your-thing`. The `-u` only on the first push; later pushes are just `git push`.
5. **Open a pull request** on GitHub/GitLab/Bitbucket. Describe what changed and why. Link the issue.
6. **Address review feedback** by making more commits on the same branch and pushing again. The PR updates automatically.
7. **Merge.** Once approved and CI is green, click merge. Most teams use a merge commit or a **squash merge** (collapses your branch into one tidy commit on `main`).
8. **Delete the branch** locally (`git branch -d feature/your-thing`) and on the remote (the PR UI has a button).

### "Push got rejected" ,  what to do

```
! [rejected]        feature/login -> feature/login (non-fast-forward)
```

Translation: someone (or you, from another machine) pushed to this branch since you last fetched. The fix:

```bash
git pull --rebase    # bring in their commits, replay yours on top
# resolve any conflicts (see above)
git push
```

`--rebase` keeps history clean ,  instead of producing a merge bubble, it replays your commits on top of theirs.

### Force-pushing ,  when it's allowed, when it's a sin

`git push --force` rewrites the remote branch to match yours. It will silently delete commits other people pushed. **Never** force-push to `main` (or any shared branch). Even on your own feature branch, prefer:

```bash
git push --force-with-lease
```

This refuses to push if the remote has commits you haven't fetched ,  a safety net that prevents you from clobbering a teammate who happened to push at the same time.

## Undoing things (safely)

Git's undo commands look similar but operate on different trees. Knowing which one you want avoids most disasters.

- `git restore <file>` ,  throw away unstaged edits to a file.
- `git restore --staged <file>` ,  unstage, but keep your edits.
- `git commit --amend` ,  replace the last commit (only before pushing).
- `git revert <hash>` ,  make a new commit that undoes an old one. Safe on shared history.
- `git reset --hard <hash>` ,  move the branch back and discard changes. **Destructive.**

> Rule of thumb: if the commit is already pushed, prefer `revert`. `reset --hard` on shared branches will make a teammate cry.

## Cheat sheet, sorted by usage

Roughly ordered from "you'll type this hourly" to "you'll need this a few times a year." The same five commands carry most of the work.

| Command | What it does | Notes |
|---|---|---|
| `git status` | Show what's changed and what's staged. | Run it constantly. It costs nothing. |
| `git add <path>` | Stage a file (or folder) for the next commit. | `git add -p` stages by hunk. |
| `git commit -m "..."` | Save staged changes as a snapshot. | Write *why*, not what. |
| `git push` | Upload commits to the remote. | First push of a new branch: `-u origin <name>`. |
| `git pull` | Fetch from remote and merge into current branch. | `--rebase` avoids merge bubbles. |
| `git log` | Show commit history. | Try `--oneline --graph --all`. |
| `git diff` | Show unstaged changes. | `--staged` shows what's about to commit. |
| `git switch <branch>` | Move to another branch. | Add `-c` to create it. |
| `git branch` | List branches (or create/delete with flags). | `-d` deletes, `-D` force-deletes. |
| `git merge <branch>` | Combine another branch into the current one. | Creates a merge commit when histories diverge. |
| `git restore <file>` | Discard local edits to a file. | `--staged` unstages instead. |
| `git clone <url>` | Copy a remote repo to your machine. | You do this exactly once per repo. |
| `git fetch` | Download remote commits without merging. | Safe ,  never changes working files. |
| `git stash` | Shelve uncommitted changes for later. | `git stash pop` brings them back. |
| `git rebase <base>` | Replay your commits on top of another branch. | Don't rebase shared history. |
| `git reset <hash>` | Move the branch pointer (and optionally the trees). | `--hard` is destructive. |
| `git revert <hash>` | Create a new commit that undoes an old one. | The safe way to undo shared commits. |
| `git remote -v` | Show configured remotes and their URLs. | Use `add` / `set-url` to edit. |
| `git tag <name>` | Mark a commit (usually a release). | Push tags with `git push --tags`. |
| `git cherry-pick <hash>` | Apply one specific commit onto your branch. | Handy for hotfixes across branches. |
| `git blame <file>` | Show who last changed each line, and when. | Less accusatory than it sounds. |
| `git init` | Turn a folder into a Git repo. | You do this once per new project. |
| `git config --global ...` | Set your name, email, default editor, aliases. | Run once when you set up a machine. |

## A few rules of thumb

- Commit early, commit small. A commit should do *one* thing.
- Read `git status` before `git add`, and `git diff --staged` before `git commit`.
- Never `--force` push to `main`. If you must rewrite, prefer `--force-with-lease` on your own branch.
- If a command scares you, run `git status` first. Git is harder to break than it looks ,  almost everything is recoverable from `git reflog`.

## Takeaway

Git is three trees and a chain of snapshots. The everyday loop is `status → add → commit → push`. Branches are movable labels; merging and rebasing are two ways to bring them back together. Everything else in the cheat sheet is a refinement of that.
