## The mental model: three trees

Almost everything in Git makes sense once you see it as three places your files can live. The working directory is what you edit. The staging area (also called the *index*) is the draft of your next snapshot — files you've marked as "include this." The repository is the permanent history of snapshots (`commits`), stored inside the `.git` folder.

## The everyday loop

Ninety percent of Git use is the same three-step cycle: see what's changed, stage what you want to keep, commit it with a message that explains *why*.

```bash
# 1. What changed?
git status

# 2. Stage the parts you want in the next snapshot
git add README.md src/parser.py

# 3. Save the snapshot with a message
git commit -m "Parser: handle empty input rows"
```

`git status` is the most under-used safety net in Git. Read it before every `add` and every `commit` — it tells you exactly what's about to move between trees.

## Commits are snapshots in a chain

A commit isn't a diff — it's a full snapshot of your project with a pointer to its parent. The history is just a chain of those snapshots. A **branch** is a movable label pointing at one of them. `HEAD` is the label that says "you are here."

## Branching off, merging back

Branches let you work without disturbing `main`. Create one, commit on it, then bring it back. Most teams call the long-lived line `main` and the short-lived ones *feature branches*.

```bash
git switch -c feature/login    # create + switch
# ...edit, add, commit...
git switch main
git merge feature/login        # bring it back
git branch -d feature/login    # delete the label
```

## Merge vs. rebase

Both bring branches together. **Merge** preserves history exactly as it happened, with a merge commit. **Rebase** rewrites your branch so its commits sit on top of the new base — a straight line, no merge bubble.

## Working with remotes

A *remote* is just another copy of the repo, usually on GitHub. The default name is `origin`. Three commands move work between you and it:

- `git fetch` — download new commits, don't touch your branch.
- `git pull` — fetch + merge into your current branch.
- `git push` — send your commits up.

## Undoing things (safely)

Git's undo commands look similar but operate on different trees. Knowing which one you want avoids most disasters.

- `git restore <file>` — throw away unstaged edits to a file.
- `git restore --staged <file>` — unstage, but keep your edits.
- `git commit --amend` — replace the last commit (only before pushing).
- `git revert <hash>` — make a new commit that undoes an old one. Safe on shared history.
- `git reset --hard <hash>` — move the branch back and discard changes. **Destructive.**

> Rule of thumb: if the commit is already pushed, prefer `revert`. `reset --hard` on shared branches will make a teammate cry.

## Cheat sheet, sorted by usage

Roughly ordered from "you'll type this hourly" to "you'll need this a few times a year." The same five commands carry most of the work.

| Command | What it does | Notes |
|---|---|---|
| `git status` | Show what's changed and what's staged. | Run it constantly. It costs nothing. |
| `git add <path>` | Stage a file (or folder) for the next commit. | `git add -p` stages by hunk. |
| `git commit -m "..."` | Save staged changes as a snapshot. | Write *why*, not what. |
| `git push` | Upload commits to the remote. | First push of a new branch: `-u origin <name>`. |
| `git pull` | Fetch from remote and merge into current branch. | `--rebase` avoids merge bubbles. |
| `git log` | Show commit history. | Try `--oneline --graph --all`. |
| `git diff` | Show unstaged changes. | `--staged` shows what's about to commit. |
| `git switch <branch>` | Move to another branch. | Add `-c` to create it. |
| `git branch` | List branches (or create/delete with flags). | `-d` deletes, `-D` force-deletes. |
| `git merge <branch>` | Combine another branch into the current one. | Creates a merge commit when histories diverge. |
| `git restore <file>` | Discard local edits to a file. | `--staged` unstages instead. |
| `git clone <url>` | Copy a remote repo to your machine. | You do this exactly once per repo. |
| `git fetch` | Download remote commits without merging. | Safe — never changes working files. |
| `git stash` | Shelve uncommitted changes for later. | `git stash pop` brings them back. |
| `git rebase <base>` | Replay your commits on top of another branch. | Don't rebase shared history. |
| `git reset <hash>` | Move the branch pointer (and optionally the trees). | `--hard` is destructive. |
| `git revert <hash>` | Create a new commit that undoes an old one. | The safe way to undo shared commits. |
| `git remote -v` | Show configured remotes and their URLs. | Use `add` / `set-url` to edit. |
| `git tag <name>` | Mark a commit (usually a release). | Push tags with `git push --tags`. |
| `git cherry-pick <hash>` | Apply one specific commit onto your branch. | Handy for hotfixes across branches. |
| `git blame <file>` | Show who last changed each line, and when. | Less accusatory than it sounds. |
| `git init` | Turn a folder into a Git repo. | You do this once per new project. |
| `git config --global ...` | Set your name, email, default editor, aliases. | Run once when you set up a machine. |

## A few rules of thumb

- Commit early, commit small. A commit should do *one* thing.
- Read `git status` before `git add`, and `git diff --staged` before `git commit`.
- Never `--force` push to `main`. If you must rewrite, prefer `--force-with-lease` on your own branch.
- If a command scares you, run `git status` first. Git is harder to break than it looks — almost everything is recoverable from `git reflog`.

## Takeaway

Git is three trees and a chain of snapshots. The everyday loop is `status → add → commit → push`. Branches are movable labels; merging and rebasing are two ways to bring them back together. Everything else in the cheat sheet is a refinement of that.
